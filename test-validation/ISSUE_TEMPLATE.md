### 问题描述示例

#### 示例 1：脚本兼容性问题

**标题**：Windows Git Bash 下 sed -i 命令失败

**问题描述**：
在 Windows Git Bash 环境下，脚本使用 `sed -i` 命令时报错，导致状态更新失败。

**重现步骤**：
1. 在 Windows Git Bash 中执行 `/comet-build`
2. 观察到状态更新阶段失败
3. 查看日志显示 `sed: -i may not be used with stdin` 错误

**期望行为**：
状态应该成功更新，脚本在所有平台兼容。

**实际行为**：
脚本在 Windows Git Bash 下失败，状态未更新。

**根因分析**：
GNU sed 和 BSD sed 的 `-i` 选项行为不同，在 Windows Git Bash 中使用不兼容。

**修复方案**：
使用 `awk` 替代 `sed -i`，或检测平台并使用相应的命令。

#### 示例 2：状态不一致

**标题**：.comet.yaml 状态与实际文件状态不一致

**问题描述**：
在 verify 阶段，`.comet.yaml` 中 `verify_result` 显示为 `pass`，但实际验证报告文件不存在。

**重现步骤**：
1. 执行完整工作流到 verify 阶段
2. 验证报告未生成，但状态更新为 `pass`
3. 检查 `.comet.yaml` 和实际文件状态

**期望行为**：
状态应该与文件状态一致，缺少验证报告时不应该更新为 `pass`。

**实际行为**：
状态提前更新，导致状态不一致。

**根因分析**：
Guard 检查未正确验证验证报告文件的存在性。

**修复方案**：
增强 Guard 检查，确保 `verification_report` 字段指向的文件必须存在。

#### 示例 3：用户决策点跳过

**标题**：build_mode 选择点未使用 AskUserQuestion 工具

**问题描述**：
在 build 阶段的 plan-ready 暂停点，应该询问用户选择 build_mode，但仅输出文字提示后自动继续。

**重现步骤**：
1. 执行 `/comet-build`
2. Plan 生成后应该暂停询问 build_mode
3. 实际仅输出文字，未等待用户选择

**期望行为**：
应该使用 AskUserQuestion 工具暂停并等待用户选择。

**实际行为**：
仅文字提示，自动使用默认值继续。

**根因分析**：
Skill 实现未正确使用 AskUserQuestion 工具。

**修复方案**：
修改 Skill 实现，在决策点必须使用 AskUserQuestion 工具。
